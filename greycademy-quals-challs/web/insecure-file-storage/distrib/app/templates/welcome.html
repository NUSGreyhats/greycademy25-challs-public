This application may be vulnerable to file/directory traversal. An attacker could manipulate file paths (for example using ../) to access files and directories outside of the intended folder.

Example: If the application reads a file using a URL parameter like ?file=report.txt, an attacker might try a value like ?file=../../etc/passwd to read sensitive system files.

Why This Is Dangerous

  • Unauthorized access to configuration files and source code
  • Exposure of passwords, API keys, and other secrets
  • Leakage of user data and system information

How to Reduce the Risk

  • Never trust user input for file paths.
  • Use whitelists of allowed files or identifiers.
  • Normalize and validate paths before using them on the server.
  • Restrict file access to a specific, non-sensitive directory.

If you are an administrator or developer, review your code wherever user input is used to build file paths and apply proper validation and access controls.
